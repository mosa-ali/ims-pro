# Humanitarian Finance Operating Model

**Purpose**: How the finance system operates in practice  
**Applies to**: Phases 2–12; organizational readiness  

---

## Finance Organization Structure

```
CFO (Chief Financial Officer)
├─ Finance Manager (GL, budget, reporting)
│  ├─ GL Specialist (GL posting, reconciliation)
│  ├─ Budget Analyst (budget allocation, variance)
│  └─ Reporting Coordinator (reports, dashboards)
├─ Treasurer (cash, payments, banking)
│  ├─ Payment Officer (payment releases, banking)
│  └─ Cash Analyst (forecasts, liquidity)
├─ Compliance Officer (rules, audit, violations)
│  └─ Compliance Analyst (monitoring, reporting)
└─ Finance Administrator (user management, system admin)
```

## Daily Operating Rhythm

### **Morning (08:00-11:00)**
```
08:00 - Finance Team Standup
├─ Previous day's GL postings: ✓/✗
├─ Current day's priorities
├─ Exceptions/violations detected
└─ Urgent actions needed

09:00 - Invoice Processing Wave
├─ Finance Manager approves pending invoices (queued overnight)
├─ 3-way match validation (auto 95%, manual 5%)
├─ GL entries posted automatically
├─ Treasurerforecasts updated

11:00 - Budget Check-In
├─ Review yesterday's spending vs. budget
├─ Flag any variances >10%
├─ Run budget availability check
└─ Notify project managers if at-risk
```

### **Afternoon (13:00-17:00)**
```
13:00 - Payment Processing
├─ Treasurer releases scheduled payments
├─ Verify cash position vs. payables due
├─ Post GL entries (GL Service)
├─ Update treasury forecast

14:00 - Risk Evaluation (Async)
├─ Financial Intelligence evaluates all day's events
├─ Risk scores updated
├─ Alerts generated (if critical)
└─ Recommendations cached

15:00 - Reconciliation
├─ Bank reconciliation (if new statements)
├─ GL account balance review
├─ Variance investigation
└─ Period reconciliation status update

16:00 - Compliance Check
├─ Review rule violations (real-time)
├─ Escalate any critical violations
├─ Monitor donor reporting deadlines
└─ Update compliance dashboard

17:00 - End-of-Day Report
├─ Daily summary: GL posted, payments released, risks
├─ Exceptions for next-day follow-up
├─ Digital Twin updated (refresh dashboards)
└─ Finance team sign-off
```

## Monthly Operating Rhythm

### **Week 1: Transactions & Processing**
```
Normal operations
├─ Invoice approvals
├─ Payment releases
├─ Budget allocations
└─ GL posting

No special activities (continue normal pace)
```

### **Week 2: Mid-Month Review**
```
Budget Review
├─ Budget utilization YTD
├─ Forecast updates
├─ Variance analysis (>20% flagged)
└─ Budget reallocation requests (if needed)

Cash Position Review
├─ 30/60/90-day forecast updated
├─ Liquidity risk assessment
├─ Payment timing optimization
└─ Bank account optimization
```

### **Week 3: Reconciliation & Reporting**
```
Bank Reconciliation
├─ Reconcile all bank statements (vs. GL)
├─ Resolve outstanding items
├─ Post timing differences
└─ Investigate variances

Vendor Reconciliation
├─ Reconcile vendor statements (vs. payables)
├─ Approve/dispute discrepancies
└─ Update vendor accounts

Reporting Preparation
├─ Run monthly reports
├─ Draft budget vs. actual analysis
├─ Prepare finance narrative (for management)
└─ Quality review
```

### **Week 4: Period Close**
```
Month-End Close Process

Day 1: Transactions Cut-Off
├─ Stop all GL entries after midnight
├─ Finalize period status
└─ Lock period (no new entries)

Days 2-3: Accruals & Adjustments
├─ Generate accruals (depreciation, interest, etc.)
├─ Generate reversals (for next month)
├─ Allocate shared costs
└─ Review adjustments

Day 4: Final Reconciliation
├─ Final GL balance verification
├─ Bank reconciliation final
├─ Trial balance validation
└─ Budget vs. actual final

Day 5: Financial Statements & Reporting
├─ Generate financial statements (Balance Sheet, P&L, CF)
├─ Prepare variance explanations
├─ Draft executive summary
├─ Prepare donor reports

Day 6: Review & Approval
├─ Finance Manager reviews statements
├─ CFO approves statements
├─ Grant managers review grant financials
└─ Stakeholder notifications

Day 7: Publishing & Distribution
├─ Publish to external reporting portals (if required)
├─ Email statements to stakeholders
├─ Archive reports (for audit)
└─ Move forward to next month

Close time target: <2 days
```

## Escalation Procedures

### **Budget Overrun** (>20% over allocation)
```
Step 1: Detection (automatic alert)
├─ Finance system flags: Project X at 105%

Step 2: Investigation (Finance Manager)
├─ Review actual vs. budget
├─ Identify reason for overrun
├─ Assess impact on grant

Step 3: Resolution Options
├─ Option A: Reallocate from another project (requires approval)
├─ Option B: Reduce scope (project manager decision)
├─ Option C: Request supplementary allocation (if donor allows)

Step 4: Approval (if reallocate)
├─ Finance Manager approves + logs reason
├─ Audit trail preserved
└─ Update budget

Step 5: Communication
├─ Notify project manager
├─ Update dashboard
└─ Document in monthly report
```

### **Liquidity Stress** (Cash <30 days of operations)
```
Step 1: Alert (automatic, when <45 days forecast)
├─ Finance system sends alert to Treasurer

Step 2: Assessment (Treasurer)
├─ Verify forecast accuracy
├─ Identify timing mismatch (payments vs. revenue)
├─ Calculate shortfall amount & timeline

Step 3: Mitigation Options
├─ Option A: Delay vendor payments (negotiate)
├─ Option B: Request grant draw-down (accelerate)
├─ Option C: Short-term loan/line of credit

Step 4: Execution
├─ Treasurer implements plan
├─ CFO approval for non-standard actions
└─ Update forecast

Step 5: Monitoring
├─ Daily cash monitoring until resolved
├─ Update executive team
└─ Post-mortem (prevent future occurrence)
```

### **Compliance Violation** (Rule broken)
```
Step 1: Detection (real-time, system blocks GL post)
├─ System identifies violation
├─ GL entry blocked (not posted)

Step 2: Notification (immediate)
├─ Alert sent to Compliance Officer
├─ Email to Finance Manager

Step 3: Investigation (Compliance Officer)
├─ Review violation
├─ Determine root cause
├─ Assess impact on donor relationship

Step 4: Resolution Options
├─ Option A: Request donor waiver (if legitimate reason)
├─ Option B: Correct procurement (restart process)
├─ Option C: Remediate violation (if possible)

Step 5: Approval & Posting
├─ If waiver approved: Post GL with waiver reference
├─ If corrected: Reprocess transaction correctly
├─ If remediated: Document and post

Step 6: Reporting
├─ Log violation in compliance dashboard
├─ Add to compliance report
├─ Add to audit trail
└─ Post-mortem for process improvement
```

## Training & Onboarding

### **Finance User Onboarding (1 week)**
```
Day 1: System orientation
├─ Walk-through finance module
├─ Invoice submission workflow
├─ Budget inquiry
└─ Dashboard overview

Day 2: Hands-on practice (sandbox)
├─ Submit sample invoice
├─ Check budget availability
├─ View payment status
└─ Generate sample report

Day 3: Real work with supervision
├─ Finance Manager watches user submit invoice
├─ Provide feedback
└─ Verify understanding

Day 4-5: Independent work
├─ User submits invoices independently
├─ Finance Manager spot-checks
└─ User certified (if competent)
```

### **Finance Manager Onboarding (2 weeks)**
```
Week 1: Deep dive
├─ GL posting process
├─ Budget management & reallocation
├─ Period close procedures
├─ Reconciliation process

Week 2: Operations & escalation
├─ Handling exceptions & violations
├─ Risk assessment & alerts
├─ Reporting & analysis
├─ Cross-module integration (Procurement, HR, etc.)

Certification: Pass competency test (GL posting, recon, close)
```

## Support Model

### **Help Desk**
```
Tier 1: User Issues (<5 min resolution)
├─ "How do I submit an invoice?"
├─ "Where do I find the budget report?"
├─ "How do I approve a payment?"
└─ "Password reset"

Response SLA: <2 hours
Escalation: If not resolvable in Tier 1
```

### **Finance Manager**
```
Tier 2: Technical Issues (5-30 min resolution)
├─ "Invoice not matching PO - why?"
├─ "How do I reallocate budget?"
├─ "GL balance doesn't reconcile"
└─ "Report calculation seems wrong"

Response SLA: <4 hours
Escalation: If not resolvable in Tier 2
```

### **System Administrator**
```
Tier 3: System/Configuration Issues (>30 min resolution)
├─ Database corruption
├─ Rule configuration errors
├─ Performance degradation
└─ System downtime

Response SLA: <1 hour (critical), <24 hours (non-critical)
Escalation: If not resolvable in Tier 3
```
