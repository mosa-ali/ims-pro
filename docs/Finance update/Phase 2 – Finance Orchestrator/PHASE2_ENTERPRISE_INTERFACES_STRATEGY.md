Orchestrator depends on interfaces, not implementations.\n * \n *   FinanceOrchestrator (Phase 2)\n *   └── Only coordinates. (Phase 3)\n *   └── Recommends actions\n *   └── Evaluates proposals\n *   └── Simulates outcomes (what-if analysis)\n *   \n *   DomainEventBus (Phase 3)\n *   └── Publishes domain events\n *   └── 3)\n *   └── Records audit trail\n *   └── Verifies integrity\n *   └── Compares state changes\n *   └── Explains decisions\n *   \n *   NotificationGateway (Phase 3)\n *   └── Email, SMS, Teams, Slack, WhatsApp, Push\n *   └── Subscribes to domain events\n *   \n *   WorkflowEngine (Phase 4+)\n *   └── Complex approval workflows\n *   └── Budget cycle management\n *   └── Procurement Services are loosely coupled (depend on interfaces)\n * ✅ Rules are configurable (not hardcoded)\n * ✅ Ready for CQRS + Event Sourcing\n * ✅ Each service can be developed/deployed independently\n * ✅ Easy to test each service in isolation\n * \n * \n * FILE STRUCTURE: 09_FinanceOrchestratorInterfaces.ts\n * ════════════════════════════════════════════════════════════════════════════\n */\n\n/**\n * SECTION 1: Financial Command/Query Pattern (CQRS Preparation)\n * ──────────────────────────────────────────────────────────────\n * \n * Instead of passing raw DTOs:\n *   postGL({ entryDate, description, lines, ... })\n * \n * Use Commands:\n *   new PostJournalCommand({ ... }).execute()\n * \n * Benefits:\n * ✅ Clear intent (PostJournalCommand vs. UpdateJournalCommand)\n * ✅ Immutable command objects (good for event sourcing)\n * ✅ Command handlers can be versioned\n * ✅ Audit trail includes commands (not just events)\n * ✅ Commands + Events = Complete CQRS pattern\n * \n * Defined interfaces:\n *   IFinancialCommand (base for all commands)\n *   IPostJournalCommand\n *   IAllocateBudgetCommand\n *   IIssueAdvanceCommand\n *   ILiquidateAdvanceCommand\n *   IProcessPaymentCommand\n *   \n *   IFinancialQuery (base for all queries)\n *   IFinancialQueryResult<T> (typed query results)\n */\n\n/**\n * SECTION 2: Domain Event Bus Interface\n * ──────────────────────────────────────\n * \n * Defined in: IDomainEventBus\n * \n * Methods:\n *   publish(eventType, payload, context)\n *     ↓ Publishes immutable event\n *     → Returns eventId for tracking\n *   \n *   subscribe(eventPattern, handler, options)\n *     ↓ Handler called for matching events\n *     → Returns subscriptionId\n *   \n *   unsubscribe(subscriptionId)\n *     ↓ Stops listening to events\n *   \n *   replay(filters, handler)\n *     ↓ Re-process events for time-travel queries\n *     → Used for debugging, testing, state reconstruction\n *   \n *   deadLetter(eventId, reason, context)\n *     ↓ Send failed events to dead letter queue\n *     → Manual inspection and retry\n *   \n *   retry(eventId, maxAttempts, backoffMs)\n *     ↓ Automatic retry with exponential backoff\n *   \n *   snapshot(snapshotId, stateHash, metadata)\n *     ↓ Performance optimization (skip replaying all events)\n * \n * Phase 3 Implementation Examples:\n * - RabbitMQ (pub/sub, dead letter queues)\n * - Kafka (event streaming, snapshots)\n * - AWS SNS/SQS (cloud-native)\n * - Apache Pulsar (multi-tenancy ready)\n */\n\n/**\n * SECTION 3: Event Store Interface\n * ─────────────────────────────────\n * \n * Defined in: IEventStore\n * \n * Methods:\n *   append(streamId, eventType, payload, metadata)\n *     ↓ Add immutable event (never updates/deletes)\n *     → Returns eventId + version\n *   \n *   load(streamId, options)\n *     ↓ Get all events for a stream\n *     → Used for state reconstruction\n *   \n *   stream(filters, onEvent)\n *     ↓ Tail event stream (like kafka consumer)\n *     → Real-time event processing\n *   \n *   replay(streamId, reducer, initialState)\n *     ↓ Reconstruct current state from events\n *     → Function passed as reducer (reducer pattern)\n *   \n *   checkpoint(consumerId, streamId, eventId, offset)\n *     ↓ Mark consumption progress\n *     → Resume from checkpoint on restart\n *   \n *   snapshot(streamId, stateData, version)\n *     ↓ Store point-in-time snapshot\n *     → Skip replaying from beginning\n * \n * Phase 3+ Implementation Examples:\n * - PostgreSQL (with custom event table)\n * - EventStoreDB (specialized event database)\n * - MongoDB (flexible document storage)\n * - DynamoDB (serverless, scaling)\n * \n * Foundation for Event Sourcing:\n *   Every state change recorded as immutable event.\n *   Current state = Reduce(events, initialState).\n */\n\n/**\n * SECTION 4: Rule Engine Interface\n * ──────────────────────────────────\n * \n * Defined in: IRuleEngine\n * \n * KEY PRINCIPLE:\n * FinanceOrchestrator NEVER hardcodes business rules.\n * \n * Instead:\n *   decision = ruleEngine.evaluate('budget_rules', context)\n *   if (decision.decision loadRules(rulesetId, org)\n *     ↓ Load rules (from DB, file, API)\n *   \n *   reloadRules(rulesetId, org)\n *     ↓ Refresh rules (pick up changes without restart)\n *   \n *   validate(rulesetId, org)\n *     ↓ Check rules for syntax/logic errors\n *   \n *   explain(rulesetId, context, org, ou)\n *     ↓ Explain WHY decision was made\n *     → Human-readable trace of rule evaluation\n *     → Essential for compliance (\"Why was this rejected?\")\n * \n * Rules Examples (configurable, not hardcoded):\n *   Donor Rules:\n *     IF donor.allowedOverhead < 10%\n *     THEN reject budget */\n\n/**\n * SECTION 5: Audit Engine Interface\n * ──────────────────────────────────\n * \n * Defined in: IAuditEngine\n * \n * Purpose: Immutable audit trail (separate from event log)\n * \n * Phase 2 Pattern:\n *   publishEvent('GL_POSTED', { ... })\n *   ↓ Done. Immutable audit trail.\n *   ↓ Can replay, compare, verify.\n * \n * Methods:\n *   record(entityType, entityId, action, before, after, context)\n *     ↓ Record what changed, who changed it, why\n *   \n *   replay(entityType, entityId, fromDate, toDate)\n *     ↓ See complete change history\n *     → Timeline of all modifications\n *   \n *   compare(entityType, entityId, timestamp1, timestamp2)\n *     ↓ What changed between two points?\n *     → Difference report\n *   \n *   verify(entityType, org, ou)\n *     ↓ Check audit trail for gaps/tampering\n *     → Compliance validation\n * \n * Use Cases:\n * - Compliance reporting (\"Who approved this and when?\")\n * - Forensic analysis (\"What was the sequence of changes?\")\n * - Change tracking (\"What changed in this budget?\")\n * - User accountability (\"Who made which changes?\")\n */\n\n/**\n * SECTION 6: Decision Engine Interface\n * ──────────────────────────────────────\n * \n * Defined in: IDecisionEngine\n * \n * Purpose: Recommend, evaluate, simulate decisions\n * \n * Different from RuleEngine:\n *   RuleEngine: \"Does this comply?\" → Yes/No\n *   DecisionEngine: \"Should we do this?\" → Recommend/Evaluate/Simulate\n * \n * Methods:\n *   recommend(scenario, context, org, ou)\n *     ↓ \"What should we do?\"\n *     → Recommendation + confidence + alternatives\n *   \n *   evaluate(scenario, proposal, criteria, org, ou)\n *     ↓ \"How good is this proposal?\"\n *     → Score + rating + improvements\n *   \n *   simulate(scenario, proposal, timeHorizon, org, ou)\n *     ↓ \"What if we do this?\"\n *     → Projected outcomes + risks + opportunities\n * \n * Example:\n *   decision = decisionEngine.simulate(\n *     'advance_issuance',\n *     { amount: 5000, currency: 'USD' },\n *     '3_months'\n *   )\n *   \n *   → Outcomes:\n *      -
